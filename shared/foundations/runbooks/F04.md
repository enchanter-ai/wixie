# F04 — Task drift

Failure code from `../conduct/failure-modes.md`.

## Signature
Work expanded past the stated goal.

## Detection
- **Audit-trail signal:** files edited or commands run that do not map to any clause of the original task.
- **Metrics signal:** diff-line count exceeds 2-3x the median for similar tasks; runtime exceeds the budgeted step count.
- **Self-report signal:** "while I was at it…", "I also fixed…", "let me also clean up…".

## Triage
1. Re-read the success criterion from the original turn.
2. List every artifact touched; mark each as in-scope or drift.
3. Confirm whether the drift was authorized (explicit user override per `../conduct/discipline.md` § When to violate).

## Rollback
- **If during a skill run:** revert drift edits; keep only changes that map to the success criterion.
- **If during a deployed plugin:** stash the wider diff; reland only the in-scope subset; file the drift as a separate task.
- **State to verify before resuming:** the diff contains only lines that satisfy (a) the change requested, (b) lines required by (a), or (c) flagged pre-existing drift.

## Post-incident
- Log the incident in the relevant plugin's `state/precedent-log.md`.
- If the failure was missed by an existing detector, file an issue under `agent-foundations/runbooks/improvements/`.
- If 3+ instances of this code in 30 days, escalate to the project's failure-log review. After three F04 in one prompt, the success criterion is vague — escalate to the developer per `failure-modes.md` § How to read the log.
