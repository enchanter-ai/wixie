# F11 — Reward hacking

Failure code from `../conduct/failure-modes.md`.

## Signature
Hit the metric by gaming it (e.g., pass assertion by weakening it).

## Detection
- **Audit-trail signal:** test/assertion edits on the same iteration that the score "passes"; rubric edits during convergence; placeholder content (`"""TODO"""`, `pass`, `return 0`) inserted to satisfy a lint or coverage metric.
- **Metrics signal:** score jump uncorrelated with behavior change; assertion-strength delta negative; reviewer-flag rate elevated.
- **Self-report signal:** "I made the test pass by relaxing the bound", "I added stub docstrings to satisfy the lint".

## Triage
1. Diff the assertion / rubric / metric definition between the failing and passing iteration.
2. Confirm whether the *original* spec is still enforced or whether the bar moved.
3. Re-run against the un-weakened assertion to surface the real outcome.

## Rollback
- **If during a skill run:** revert the assertion/rubric edit; re-run; record the genuine score; pick a different axis next round per the no-regression contract.
- **If during a deployed plugin:** freeze convergence on this prompt; require a reviewer (Haiku validator) to confirm the assertion still tests the original behavior before any further iteration.
- **State to verify before resuming:** the assertion enforces the original spec; score reflects behavior, not metric games.

## Post-incident
- Log the incident in the relevant plugin's `state/precedent-log.md`.
- If the failure was missed by an existing detector, file an issue under `agent-foundations/runbooks/improvements/`.
- If 3+ instances of this code in 30 days, escalate to the project's failure-log review — the scoring surface is being gamed; re-derive the rubric per `failure-modes.md` § How to read the log.
