# F09 — Parallel race

Failure code from `../conduct/failure-modes.md`.

## Signature
Two writes to the same file / same branch.

## Detection
- **Audit-trail signal:** two parallel `Write`/`Edit`/`Bash`-mutating calls in one assistant turn targeting the same path; JSONL files with interleaved or partial lines.
- **Metrics signal:** corrupt JSON parse rate on append-only logs; lost-update rate detected at the next reconcile.
- **Self-report signal:** "let me run these two writes in parallel…" without a partition strategy.

## Triage
1. Identify all write/mutation events in the same dispatch window; check whether they targeted the same path or branch.
2. Inspect the affected file for interleaved content (JSONL: orphan partial lines; text: half-overwritten regions).
3. Determine whether one write's payload was lost or merged incorrectly.

## Rollback
- **If during a skill run:** revert the affected file to its pre-race baseline; re-apply the writes serially in dependency order.
- **If during a deployed plugin:** restore from baseline; switch the call site to a locked append helper (`fcntl.flock` / `msvcrt.locking`) per spec G; or partition writes by path.
- **State to verify before resuming:** the affected file parses cleanly; no partial lines; all intended payloads present.

## Post-incident
- Log the incident in the relevant plugin's `state/precedent-log.md`.
- If the failure was missed by an existing detector, file an issue under `agent-foundations/runbooks/improvements/`.
- If 3+ instances of this code in 30 days, pause the plugin — contract broken. Escalate to the project's failure-log review.
