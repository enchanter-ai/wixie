# F07 — Over-helpful substitution

Failure code from `../conduct/failure-modes.md`.

## Signature
Solved a problem the user didn't ask about.

## Detection
- **Audit-trail signal:** edits to files outside the named scope; commits or diffs that include "while I was here" reformatting.
- **Metrics signal:** diff scope/task scope ratio > 1.5; review-comment rate elevated on PRs from this skill.
- **Self-report signal:** "I noticed and fixed…", "I also improved…", "I cleaned up the imports while…".

## Triage
1. List every changed file/region; for each, ask: did the user ask for this?
2. Mark drift edits; separate from in-scope edits.
3. Confirm whether the user explicitly opened scope ("clean up this file") — if not, the substitution is unsolicited.

## Rollback
- **If during a skill run:** revert the unsolicited edits; keep only the asked-for change; mention the spotted issues in the reply, do not fix them silently.
- **If during a deployed plugin:** stash the wider diff; reland the surgical subset; file the cleanup as a separate task.
- **State to verify before resuming:** the diff is surgical per `discipline.md` § Surgical changes — every line maps to the request or a dependency of it.

## Post-incident
- Log the incident in the relevant plugin's `state/precedent-log.md`.
- If the failure was missed by an existing detector, file an issue under `agent-foundations/runbooks/improvements/`.
- If 3+ instances of this code in 30 days, escalate to the project's failure-log review — systemic over-helpfulness.
