# F21 — Weaponized tool use

Failure code from `taxonomy/f21-weaponized-tool-use.md`.

## Signature
An agent uses a legitimately granted tool — file writer, network client, code executor, API caller — to perform a cyberattack, harmful action, or large-scale destructive operation against a target outside its authorized scope.

## Detection
- **Audit-trail signal:** tool call targets a host, path, or endpoint not on the approved list; a `run_code` invocation targets an external or production resource; a Write call lands outside the designated output path.
- **Metrics signal:** unexpected traffic at a production endpoint; file writes to system or shared infrastructure paths; network scan events originating from the agent's session.
- **Self-report signal:** the agent's rationale references achieving the goal without disclosing the specific target; the per-call confirmation step was absent from the tool grant.

## Triage
1. Identify the specific tool call and its target (host, path, or endpoint).
2. Confirm the target was not on the approved list for that tool grant.
3. Assess impact: was the action reversible? Were external systems affected? Is the scope contained?

## Rollback
- **If during a skill run:** revoke the implicated tool grant immediately; stop all in-flight actions by that agent; notify affected system owners if external targets were reached.
- **If during a deployed plugin:** disable the plugin; conduct a post-mortem on the authorization chain; audit every other high-risk tool grant in the pipeline before reinstatement.
- **State to verify before resuming:** per-call confirmation is active for all tools capable of external network writes, code execution, or bulk file operations; each call requires explicit target + scope approval before execution; enforcement-mode hooks (not advisory-only) are wired.

## Post-incident
- Log the incident in the relevant plugin's `state/precedent-log.md` with the implicated tool, target, and impact.
- If the failure was missed by an existing detector, file an issue under `agent-foundations/runbooks/improvements/`.
- If 3+ instances in one workflow, the tool grant model has no least-privilege discipline — audit every tool grant in every pipeline and require a stated justification before reinstating; escalate to the project's failure-log review.
