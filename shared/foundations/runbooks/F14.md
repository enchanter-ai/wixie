# F14 — Version drift

Failure code from `../conduct/failure-modes.md`.

## Signature
Used deprecated API / retired model ID / old flag.

## Detection
- **Audit-trail signal:** API call against a retired endpoint; tool invocation with a removed flag; model ID not present in `shared/models-registry.json`.
- **Metrics signal:** runtime-error rate spike correlated with vendor deprecation dates; prompt outputs cite outdated SDK versions.
- **Self-report signal:** "I'll use the X API" referencing a method removed in a prior major version; emission of model IDs the registry has retired.

## Triage
1. Identify the cited identifier (model ID, API method, flag, package version).
2. Cross-check against `shared/models-registry.json` and the vendor's current docs.
3. Confirm whether a current replacement exists; if so, swap before any further generation.

## Rollback
- **If during a skill run:** halt generation; regenerate against the current registry; verify the replacement satisfies the same contract.
- **If during a deployed plugin:** bump the plugin's registry-pin to the latest version; add a registry-mismatch advisory hook (injecting, not blocking) per `../conduct/hooks.md`.
- **State to verify before resuming:** every external identifier in the artifact passes a registry check; no retired IDs remain.

## Post-incident
- Log the incident in the relevant plugin's `state/precedent-log.md`.
- If the failure was missed by an existing detector, file an issue under `agent-foundations/runbooks/improvements/`.
- If 3+ instances of this code in 30 days, audit the registry freshness per `failure-modes.md` § Escalation patterns. Escalate to the project's failure-log review.
