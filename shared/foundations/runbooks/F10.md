# F10 — Destructive without confirmation

Failure code from `../conduct/failure-modes.md`.

## Signature
`rm`, `reset --hard`, `force push` without explicit yes.

## Detection
- **Audit-trail signal:** `Bash` events containing `rm -rf`, `git reset --hard`, `git push --force`, `git branch -D`, `DROP`, `TRUNCATE` with no preceding plan-emission turn.
- **Metrics signal:** unscheduled state loss; missing files reported by user post-action; force-push events on protected branches.
- **Self-report signal:** the agent emits the destructive command directly without first listing what would be affected.

## Triage
1. Identify the destructive op and what it touched.
2. Check whether a dry-run / plan-emission turn preceded it with explicit user yes.
3. Assess blast radius: files lost, refs lost, data lost.

## Rollback
- **If during a skill run:** stop further actions immediately. Recover from git reflog (`git reflog`, `git fsck --lost-found`), filesystem snapshot, backup, or remote.
- **If during a deployed plugin:** disable the plugin until the missing dry-run gate is restored; require explicit confirmation per `../conduct/verification.md` § Dry-run for destructive ops.
- **State to verify before resuming:** lost state is recovered or the loss is acknowledged in writing; the dry-run gate is active for future invocations.

## Post-incident
- Log the incident in the relevant plugin's `state/precedent-log.md`.
- If the failure was missed by an existing detector, file an issue under `agent-foundations/runbooks/improvements/`.
- If 3+ instances of this code in 30 days, pause the plugin — contract broken. Escalate to the project's failure-log review.
